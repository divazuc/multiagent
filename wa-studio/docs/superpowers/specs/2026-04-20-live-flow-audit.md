# Live-Flow Audit & Gap Analysis
**Date:** 2026-04-20

## Goal

Before building the business-overview dashboard, understand exactly what the live conversation pipeline writes where, and identify every gap between the current state and "the dashboard can render real numbers for a business."

## Method

1. Created a demo studio business end-to-end (`tests/demo_run.mjs`).
2. Seeded `business_profiles` (with studio-shaped persona / services) and 5 answered FAQ items.
3. Sent 5 live-mode customer messages + 3 learning-mode turns through the actual n8n webhook.
4. Inspected n8n execution run-data for node-by-node inputs and outputs.
5. Counted rows in every schema.sql and schema_v2.sql table, filtered to the demo business.

Full audit output: `tests/demo_audit.json` (generated by the script above).

---

## Executive summary

**Live mode is visibly functional — but nothing gets saved.**

- Live turns return an agent response through the webhook (5/5 replied), but:
  - `conversation_messages` has **0 rows** for the demo live session. Same for learning. WA_06 (Save Conversation) is invoked, runs to completion, and explicitly returns `"Missing required fields: session_id, business_id, user_message, agent_response, stage"`. The save silently fails inside its own validator.
  - Every schema_v2 "prod" table (`prod_conversations`, `prod_messages`, `business_config`, `business_persona`, `contacts`, `business_usage_daily`, `business_usage_monthly`, `external_leads_sources`, `admin_sessions`) is **empty across the whole database** — not just for the demo business.
- The 14 existing `conversation_messages` rows in the DB are all from **setup mode only**, written directly by WA_00's `Save setup conversation` Postgres node, not by WA_06. That path works. The live/learning path writing through WA_06 has likely **never successfully saved a message**.
- Live replies come back as generic English (`"Hey there! What can I help you with today?"`) even with a complete Hebrew `business_profiles` row in place. WA_02 returns `services: []` and empty `persona: {}` to WA_03 despite the row being seeded. Separate, second bug — the profile isn't being loaded, so WA_03 has nothing to personalize on.

## Root cause: WA_06 payload mismatch

WA_00's `Save conversation state` node is an `executeWorkflow` call to WA_06 with **no input mapping**. Whatever `$json` is at call time gets passed through. The upstream (`Check validation status`) forwards WA_05's output, which has this shape:

```json
{
  "status": "success",
  "result": {
    "validated_response": "Hey there! What can I help you with today?",
    "next_stage": "clarification",
    "action": "none",
    "cta_triggered": false,
    "escalate": false,
    "escalation_reason": null,
    "validation_passed": true,
    "rewrite_applied": false
  },
  "error": null
}
```

WA_06's `Receive sub-workflow input` Set node reads `$json.session_id`, `$json.business_id`, `$json.user_message`, `$json.agent_response`, `$json.stage`. None of those exist at the top level *or* inside `result`. WA_06 detects the missing fields in its `Validate required fields` Code node and returns an error without inserting. WA_00 swallows the error.

### Why setup works and live doesn't

Setup uses a completely different path: `Save setup conversation` is a **direct Postgres node inside WA_00** with an inline query that reads `{{ $('Webhook inbound').item.json.body.message }}` for user_message, `{{ $('Extract setup inputs').item.json.* }}` for everything else. It has direct access to the upstream nodes' data. It was added later (see commit `94dbc58` / clarification-loop shipping) specifically to save stage-tagged setup turns.

WA_06 was designed earlier, expecting its caller to build and pass a normalized payload — but the caller was never updated to do so.

## Second bug — profile not carrying into WA_03

Even though `business_profiles` has the full row, the WA_02 (Load Session and Context) output observed in n8n execution run-data shows:

```json
"business_profile": { "services": [] },
"persona": {},
"guardrails": {},
"hebrew_patterns": {},
```

So WA_03 runs against empty context → generic English reply. This is independent of the save bug and also blocks a meaningful dashboard (no "tailored response rate" metric is possible if responses aren't tailored).

Not investigated further in this audit — flagged for separate diagnosis before live traffic.

## Gaps, ranked

### G1 — [BLOCKER for dashboard] Live + learning conversations don't save to `conversation_messages`
- **What's needed:** add a payload-builder step in WA_00 between `Check validation status` / `Check learning status` and `Save conversation state`. It must supply `session_id` (from `Load session and context`), `business_id` (same), `user_message` (from `Normalize message`), `agent_response` (from `result.validated_response`), `stage` (from `result.next_stage`), plus the remaining fields WA_06 expects.
- **Shape:** either a Set/Code node that transforms `{status, result}` into a flat record for WA_06, **or** replace the executeWorkflow call with the same kind of direct Postgres node that setup uses.
- **Preferred fix:** direct Postgres node (mirrors setup's proven pattern; removes the fragile subworkflow contract). Requires a new node + two rewires in WA_00.

### G2 — [BLOCKER for dashboard] `prod_conversations` / `prod_messages` never written
- **What's needed:** after (or alongside) the `conversation_messages` save, write to the prod tables. Schema_v2 was deployed for the admin/client UI but no writer exists.
- **Decision needed from user:** dual-write (keep `conversation_messages` as the legacy audit + `prod_*` as the UI source), or migrate live path to prod-only. Recommend **dual-write** for now — zero-risk to existing setup flow.

### G3 — [BLOCKER for dashboard] Usage-counter tables empty
- `business_usage_daily` / `business_usage_monthly` are the obvious source for "total conversations today / this month / all time" metrics. Nothing increments them.
- **Fix options:**
  - **Push:** WA_00 (or WA_06/new Postgres node) does a daily upsert (`INSERT … ON CONFLICT (business_id, day) DO UPDATE SET conversations_count = conversations_count + 1`).
  - **Pull:** compute on the fly from `prod_messages` GROUP BY day. Simpler, but slower at scale. Fine for MVP.
  - Recommendation: **pull on demand** until a business crosses ~1 k conversations/day, then add cached writes.

### G4 — `contacts` never populated
- For any CRM-style view (see Step 3 in the product roadmap) we need one row per unique customer phone number, created on first inbound message.
- **Fix:** upsert to `contacts` in the same node as the message write: `ON CONFLICT (business_id, phone) DO UPDATE SET last_seen_at = NOW()`.

### G5 — WA_02 doesn't load business_profile into WA_03 context
- Independent of save — affects response quality. See "Second bug" above.
- **Fix:** inspect `Load Session and Context` workflow (WA_02) + its joins with `business_profiles`. Out of scope for this audit; open as a separate spec before wiring real WhatsApp traffic.

### G6 — `admin_sessions` never populated
- Schema_v2 table. Intended purpose unclear — probably for the business-owner login to the dashboard. Needs product decision.

### G7 — `external_leads_sources` never populated
- Schema_v2 table. Per the product roadmap's second bundle (Google-Sheet lead sync), this is likely the destination for imported leads. Not a live-flow gap — queue for the lead-management spec.

### G8 — `business_config` / `business_persona` never populated
- Per schema_v2 memory: "prod tables for admin/client UI — not yet wired." These duplicate content from `business_profiles` in a normalized shape. Not blocking the dashboard if we keep reading from `business_profiles`. Recommend deferring the migration.

## Recommended fix order (for the dashboard work ahead)

1. **G1 — Save live+learning messages** (required; without this, every dashboard metric reads zero)
2. **G2 — Dual-write to `prod_conversations` / `prod_messages`** (required; the dashboard UI should read from prod tables per the original schema_v2 intent)
3. **G4 — Populate `contacts`** (required for CRM view)
4. **G5 — Fix WA_02 profile loading** (required before real WhatsApp traffic; nice-to-have for the dashboard since we can still count conversations with stale profiles)
5. **G3 — usage counters** (optional — can pull from `prod_messages` at first)
6. **G6/G7/G8** — deferred to later specs
