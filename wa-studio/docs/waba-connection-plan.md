# אפיון: חיבור הבוט ל-WhatsApp Cloud API (WABA)

**תאריך:** 2026-07-23 · **סטטוס:** אפיון לאישור, טרם מומש

שני שלבים:
- **שלב א׳ — חיבור פרטי (עכשיו):** מספר ייעודי תחת ה-WABA שלנו, ישירות מול Cloud API. בלי תלות באישורים ממטא.
- **שלב ב׳ — Coexistence (אחרי אישור Tech Provider):** לקוחות מחברים את המספר הקיים שלהם בלי לוותר על אפליקציית WhatsApp Business — ה-flow המלא של Embedded Signup.

---

## 1. מה כבר קיים בקוד (מהעבודה של מאי)

| רכיב | מצב |
|---|---|
| `GET /wa-inbound` — אימות webhook (`hub.challenge` + verify token) | ✅ קיים |
| `POST /wa-inbound` — קליטת הודעות Cloud API | ✅ קיים |
| `normalize.js` — פרסור payload של מטא (טקסט + כפתורים אינטראקטיביים), חילוץ `phone_number_id` | ✅ קיים |
| ניתוב לעסק לפי `businesses.wa_phone_number_id` (multi-tenant) | ✅ קיים (`context.js`) |
| `wa-send.js` — שליחה ב-Graph API v21, קרדנציאלים פר-עסק (`wa_phone_number_id` + `wa_access_token`) עם fallback ל-env | ✅ קיים |
| משתני סביבה ב-Railway: `WHATSAPP_ACCESS_TOKEN`, `WHATSAPP_PHONE_NUMBER_ID`, `WHATSAPP_WEBHOOK_VERIFY_TOKEN` | ✅ קיימים |
| שעות פעילות + חגי ישראל + agent_active נאכפים ב-inbound | ✅ קיים |

כלומר: השלד נכון. מה שחסר הוא ההקשחה שהופכת אותו לעמיד מול התנהגות אמיתית של מטא.

## 2. פערים קריטיים (חובה לשלב א׳)

1. **Ack מהיר + עיבוד אסינכרוני** — היום ה-webhook עונה למטא רק אחרי כל הפייפליין (12–25 שנ׳). מטא מצפה ל-200 תוך שניות; תגובה איטית ⇒ retries ⇒ **תשובות כפולות ללקוח**, ובהמשך השבתת ה-webhook. נדרש: זיהוי payload של מטא ⇒ להחזיר 200 מיד ⇒ להריץ את הפייפליין ברקע ⇒ לשלוח את התשובה עם `wa-send`. (הפורמט הישיר של הסטודיו נשאר סינכרוני כמו היום.)
2. **דדופ לפי `message.id`** — מטא שולחת retries; `normalize.js` לא מחלץ את `msg.id` בכלל. נדרש: חילוץ + טבלת/מטמון "כבר טופל".
3. **סטטוסים ו-payloads שאינם הודעה מחזירים 400** — כל webhook של delivered/read נופל היום ל-`No message found` → 400. מטא סופרת כשלים ומשביתה. נדרש: לזהות `statuses` (ובהמשך echoes) ולהחזיר 200 בשקט.
4. **אימות חתימה `X-Hub-Signature-256`** — היום כל אחד יכול לפגוע ב-endpoint. נדרש: אימות HMAC עם ה-App Secret.
5. **הודעות שאינן טקסט** (תמונה/אודיו/מיקום) — נופלות היום ל-400. נדרש: תשובת fallback מנומסת ("כרגע אני עונה רק להודעות טקסט 🤍 ") או התעלמות + 200. טיפול מלא במדיה — מחוץ לסקופ (בהתאם לפער שהוגדר מול קרופניק).
6. **חלון 24 שעות + תבניות** — מענה לפניות נכנסות תקין תוך 24 ש׳; אבל **פולו-אפים יזומים** (הפיצ׳ר קיים בקוד וממתין) דורשים תבנית מאושרת ב-WhatsApp Manager ושליחה כ-`template`. נדרש: תבנית פולו-אפ ראשונה + מסלול שליחה.

## 3. שלב א׳ — חיבור WABA פרטי

**מטרה:** מספר אחד אמיתי עונה בפרודקשן, תחת ה-Business Manager שלנו — לפיילוט/דמו חי, בלי תלות באישורי Tech Provider.

### צד מטא (הגדרה חד-פעמית, ~חצי יום)
1. אפליקציית Meta מסוג Business (קיימת? לוודא) + מוצר WhatsApp.
2. WABA + מספר ייעודי (מספר חדש/וירטואלי — לא מספר שמשמש אפליקציה בטלפון).
3. System User עם טוקן קבוע (הרשאות `whatsapp_business_messaging` + `whatsapp_business_management`).
4. הגדרת webhook: `https://wagent.divdev.co/wa-inbound` + ה-verify token מה-env, הרשמה לשדה `messages`.
5. עדכון `businesses.wa_phone_number_id` (+ טוקן פר-עסק או fallback מה-env).

### צד קוד (לפי סדר)
| # | משימה | הערכה |
|---|---|---|
| 1 | Fast-ack + עיבוד אסינכרוני ל-payload של מטא | 0.5–1 יום |
| 2 | דדופ `message.id` (מטמון + עמודה ב-agent_runs) | 0.5 יום |
| 3 | סטטוסים/לא-הודעות → 200 שקט | 0.5 יום |
| 4 | אימות חתימה X-Hub-Signature-256 | 0.5 יום |
| 5 | fallback להודעות לא-טקסטואליות | 0.5 יום |
| 6 | תבנית פולו-אפ + שליחת template מחוץ לחלון | 1–2 ימים |
| 7 | E2E: מספר בדיקה של מטא → מספר אמיתי, כולל תרחישי הדגמה | 0.5–1 יום |

**סה״כ: ~4–6 ימי עבודה.** תואם את ההערכה שניתנה בהצעה לקרופניק (3–5 ימים לחיבור).

### הגדרת סיום (DoD)
הודעה למספר האמיתי מקבלת תשובת בוט תוך <30 שנ׳; אין תשובות כפולות תחת עומס retries; סטטוסים לא מייצרים שגיאות; ליד מופיע בדשבורד הלקוח; פולו-אפ נשלח בתבנית מאושרת.

## 4. שלב ב׳ — Coexistence כ-Tech Provider

**מה זה:** אותו מספר חי גם באפליקציית WhatsApp Business בטלפון של הלקוח וגם ב-Cloud API — הבוט עונה, והצוות ממשיך לענות ידנית מהאפליקציה כרגיל. זמין עולמית מאז מאי 2025 (כולל ישראל/אירופה).

### תנאים מוקדמים (לא בשליטתנו — במקביל)
- ✳️ **אימות עסקי + אישור Tech Provider** (בתהליך מולך).
- Advanced Access להרשאות `whatsapp_business_messaging` + `whatsapp_business_management`.
- הגדרת **Embedded Signup v4** (⚠️ v2 מושבת ב-15.10.2026 — בונים v4 מהיום, לא נוגעים בגרסאות ישנות) + Facebook Login for Business.

### חוויית הלקוח (ה-flow)
1. בטאב **"הגדרות"** בדשבורד הלקוח נוסיף כפתור **"חיבור WhatsApp"**.
2. נפתח חלון Embedded Signup של מטא → הלקוח מתחבר לפייסבוק → בוחר במסלול Coexistence → **סורק QR מאפליקציית WhatsApp Business שלו** (נדרשת גרסת אפליקציה עדכנית).
3. אנחנו מקבלים code → מחליפים ל-business token → שומרים פר-עסק (`wa_phone_number_id`, `waba_id`, טוקן) → רושמים את ה-WABA ל-webhooks שלנו.
4. תוך דקות מגיעה **סנכרון היסטוריה** (webhook `history`) — אפשר לייבא שיחות עבר לדשבורד (אופציונלי, שלב ב׳2).

### שינויי ריצה מהותיים (הלב של Coexistence)
- **`smb_message_echoes`** — כל הודעה שהצוות שולח מהאפליקציה מגיעה אלינו כ-webhook. נשמור אותה בשיחה (תיוג "נשלח ידנית") — הדשבורד כבר יודע להציג את זה.
- **Human takeover:** אם הצוות ענה ידנית מהאפליקציה — הבוט **משתתק לאותה שיחה** לחלון זמן (ברירת מחדל 60 דק׳, קונפיגורבילי פר-עסק). זה הכלל שמונע מצב שהבוט "מתווכח" עם בעל העסק מול הלקוח.
- דדופ נוסף: לא לענות על echo של עצמנו.

### מגבלות ידועות (לתקשר ללקוחות מראש)
- אין תג עסק רשמי (OBA/כחול) על מספר Coexistence.
- אין מעבר/מיגרציה בין WABAs.
- הודעות מ-clients לא נתמכים (למשל WhatsApp Web ישן) לא ייצרו echoes — ייתכנו חורים בסנכרון.

### עבודה טכנית משוערת
| משימה | הערכה |
|---|---|
| קומפוננטת Embedded Signup v4 בדשבורד (SDK של פייסבוק) | 1–2 ימים |
| endpoint להחלפת code→token + שמירה פר-עסק + הרשמת webhooks | 1 יום |
| טיפול ב-echoes + human-takeover pause | 1–2 ימים |
| ייבוא היסטוריה (אופציונלי) | 1–2 ימים |
| ניהול תבניות פר-לקוח + חיובי שיחות (החלטה עסקית: על מי הקרדיט-ליין) | 1 יום |

**סה״כ: ~5–8 ימי עבודה** אחרי קבלת האישורים. שלב א׳ הוא תשתית ישירה לשלב ב׳ — שום דבר לא נזרק.

## 5. סיכונים והחלטות פתוחות

1. **אחסון טוקנים** — היום `wa_access_token` עמודה רגילה. מינימום: וידוא שאין אליה גישת anon (מכוסה ע״י מדיניות ה-proxy); בהמשך הצפנה או Vault.
2. **על שם מי ה-WABA בשלב א׳** — המלצה: שלנו (שליטה מלאה, מהיר). ללקוח אמיתי בשלב א׳ אפשר גם WABA בבעלותו עם גישת אדמין לנו — אבל עדיף לחכות ל-Coexistence.
3. **חיוב שיחות מטא** — בפיילוט על חשבוננו; ב-Coexistence צריך להחליט אם קרדיט-ליין שלנו (ואנחנו מחייבים את הלקוח) או אמצעי תשלום של הלקוח.
4. **מדיה דו-כיוונית** — מחוץ לסקופ שני השלבים; נשאר בפער-הפיתוח שסוכם מול קרופניק (3–5 ימים בנפרד).

## 6. סדר עבודה מומלץ

```
עכשיו          → שלב א׳ סעיפים 1–5 (הקשחת ה-webhook) — גם בלי מספר חדש זה משפר את הקיים
עם מספר ייעודי → שלב א׳ 6–7 + פיילוט חי
במקביל         → להמשיך את אימות ה-Tech Provider מול מטא
אחרי אישור     → שלב ב׳: ES v4 → echoes → takeover → היסטוריה
```
